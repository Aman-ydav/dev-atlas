# 19 — Coding Standards

> Companion to `08-backend-architecture.md` (where code lives) and `07-api-design.md` (the wire contract) — this document owns **how it's written**, not how it's organized or what it returns. Every rule below is extracted from patterns the codebase has already converged on, not aspirational invention; where the codebase is still inconsistent, this document declares the winner and names the stragglers to fix on next touch, not in a dedicated cleanup commit.

## 1. Naming Conventions

### 1.1 Backend file naming per layer

| Layer | Pattern | Real examples |
|---|---|---|
| Model | `<resource>.model.js` | `knowledge.model.js`, `userProgress.model.js` |
| Controller | `<resource>.controller.js` | `knowledge.controller.js`, `dashboard.controller.js` |
| Routes | `<resource>.routes.js` | `knowledge.routes.js` |
| Validator | `<resource>.validator.js` | `progress.validator.js` |
| Service | `<workflow>.service.js` | `csvImport.service.js` |
| Middleware | `<concern>.middleware.js` | `auth.middleware.js`, `rateLimiter.middleware.js` |
| Util | `<concern>.js`, no suffix | `slugify.js`, `tokens.js`, `pagination.js` |

The resource/concern segment is always **camelCase**, singular where it names one thing (`knowledge`, `category`, `userProgress`) — never kebab-case, never snake_case, never pluralized. This is a flat rule with zero documented exceptions across every file in `backend/src`; don't introduce `user-progress.model.js` or `knowledge_controller.js` because it "reads better" in isolation — it won't match anything else in the folder.

One resource, one file, per layer. `category.controller.js` and `company.controller.js` are nearly identical in shape and still separate files — don't fold a second small resource into an existing controller to save a file.

### 1.2 Mongoose model naming, singular vs. plural

- Registered model name: **PascalCase singular** — `Knowledge`, `User`, `UserProgress`, `Category`.
- Always a **named** export: `export const Knowledge = mongoose.model("Knowledge", knowledgeSchema);` — never `export default`.
- Never pass an explicit collection-name argument to `mongoose.model()`. Accept Mongoose's automatic lowercased-plural collection name in every case, including the slightly unusual `userprogresses` (from `UserProgress`) — the payoff is that the rule has **zero exceptions to remember**, not that every resulting name is pretty. Full collection list: `06-database-design.md` §1.

**REST prefix and collection name are allowed to diverge.** `/api/v1/knowledge` (mass-noun singular, mirrors the model name) routes to the `knowledges` collection; `/api/v1/progress` routes to `userprogresses`. This is intentional, not an inconsistency to fix — the URL prefix optimizes for how the resource reads in a path and in an RTK Query hook name (`useGetKnowledgeListQuery`, not `useGetKnowledgesListQuery`), the collection name optimizes for Mongoose's one uniform pluralization rule. Don't chase them into agreement.

### 1.3 Frontend naming

| Kind | Pattern | Location | Example |
|---|---|---|---|
| Component | `PascalCase.jsx`, named export | `components/<domain>/` | `components/knowledge/KnowledgeCard.jsx` → `export function KnowledgeCard(...)` |
| Page | `PascalCase` + `Page` suffix | `src/pages/`, `src/pages/admin/` | `DashboardPage.jsx`, `AdminUsersPage.jsx` |
| Route guard | `PascalCase` + `Route` suffix | `src/routes/` | `ProtectedRoute.jsx`, `AdminRoute.jsx` |
| RTK Query slice | camelCase + `Api` suffix | `store/api/` | `knowledgeApi.js` — injects endpoints into the one shared `apiSlice`, never a second `createApi` instance |
| Redux slice | camelCase + `Slice` suffix | `store/slices/` | `authSlice.js` |
| Helper module | camelCase, no suffix | `lib/` | `apiHelpers.js`, `iconMap.js`, `utils.js` |
| Hook | camelCase, `use` prefix | `hooks/` | new hand-authored hooks: `useDebounce.js` |

**One documented exception:** anything generated by the shadcn CLI — every file under `components/ui/` and `hooks/use-mobile.js` — keeps the generator's own kebab-case naming verbatim (`components/ui/dropdown-menu.jsx`, `hooks/use-mobile.js`). Don't rename these to fit the table above; a renamed file silently breaks the CLI's ability to re-sync that component later. Everything hand-authored follows the table above; everything shadcn owns follows shadcn's own convention.

Import via the `@/*` alias for anything outside the current file's own folder (`@/components/ui/card`, `@/store/slices/authSlice`, `@/src/pages/DashboardPage`); relative imports (`./`, `../`) only for same-folder siblings.

## 2. The `ApiResponse` / `ApiError` / `asyncHandler` Contract

This is not a convention with judgment calls — it's a fixed contract with **zero exceptions** anywhere in `backend/src`. No route handler calls `res.json(...)` with a bare object, and no failure path does anything but `throw new ApiError(...)`.

```js
// backend/src/utils/apiResponse.js
class ApiResponse {
    constructor(statusCode, data, message = "Success") {
        this.statusCode = statusCode;
        this.success = true;
        this.message = message;
        this.data = data;
    }
}

// backend/src/utils/apiError.js — extends Error; also carries statusCode, errors[], data:null, success:false
```

Every controller has this exact shape:

```js
const updateCompany = asyncHandler(async (req, res) => {
    const company = await Company.findOne({ _id: req.params.id, isDeleted: false });
    if (!company) throw new ApiError(404, "Company not found");

    Object.assign(company, req.body);
    await company.save();

    return res.status(200).json(new ApiResponse(200, company, "Company updated"));
});
```

Rules this example encodes for every controller in the codebase:

- Wrapped in `asyncHandler` — always, no exceptions, including admin-only and read-only handlers. This is what makes a bare `throw` safe anywhere inside (`08-backend-architecture.md` §3, §9).
- Not-found and permission failures are a `throw new ApiError(...)`, never an early `return res.status(404).json(...)`.
- Success is always `return res.status(code).json(new ApiResponse(code, data, message))` — the `message` argument is a short human-readable phrase (`"Company updated"`), not a machine-parseable code; clients key behavior off `statusCode`/`success`, never off `message` text.
- Status codes follow the fixed table in `07-api-design.md` §0 (`200`/`201`/`204`/`400`/`401`/`403`/`404`/`409`/`422`/`429`/`500`) — check that table before inventing a new one for a new resource.

## 3. Controller-Thin, Service-Aware Pattern

**Default: business logic lives directly in the controller**, including resource-specific helper functions declared in the same file (`buildFilter`, `pickTypeFields`, and the higher-order `makeListEndpoint` in `knowledge.controller.js` / `progress.controller.js` are real, current examples — none of that is considered "too much logic for a controller"). This is a deliberate project convention, not a shortcut: for a codebase this size, a `services/` file per resource would mostly be a controller with extra indirection and no second caller.

**Promote logic to `services/<workflow>.service.js` only when at least one of these is true:**

- It's genuinely **reused by two or more controllers** — extract it if/when a second caller shows up, not preemptively.
- It's a **self-contained, heavy workflow** that would dominate and obscure the controller it's called from — streaming file parsing, multi-step third-party orchestration, batch processing with its own partial-success/error-accumulation shape.

The one real instance today is `services/csvImport.service.js`: `importDsaCsv(filePath, authorId)` streams a CSV through `csv-parse`, resolves category/company slugs to ObjectIds, and builds a `{ created, skipped, errors: [...] }` report row by row. `knowledge.controller.js`'s `importDsaCsvHandler`, by contrast, is little more than "check `req.file`, call the service, wrap the result in an `ApiResponse`." That size and complexity differential — a genuine multi-step workflow versus a thin call site — is exactly the signal that justifies the extra file.

**Don't create a service just to wrap a single Mongoose call.** `getCompanies`, `createCompany`, `updateCompany`, `deleteCompany` each do exactly one `Company.*` call plus response-shaping — that stays in the controller. A `company.service.js` that just re-exports thin wrappers around `Company.find` / `Company.create` adds a file and an import with no behavior it's protecting.

**`utils/` is a third, distinct thing** — stateless helpers or thin third-party SDK wrappers with no DevAtlas business rules (`slugify`, `pagination`, `tokens`, `cloudinary`). The tell: a `utils/*.js` file never imports a Mongoose model directly. `generateUniqueSlug(model, title)` takes a model as a *parameter* precisely so `utils/slugify.js` stays generic instead of importing `Knowledge` or `Category` itself.

## 4. Mongoose Schema Conventions

**`{ timestamps: true }` on every schema, no exceptions.** All nine top-level schemas (`User`, `Category`, `Company`, `Knowledge`, `UserProgress`, `Annotation`, `Resource`, `Attachment`, `Activity`) declare it; a tenth added later does too, even if it seems purely transactional — `createdAt`/`updatedAt` are cheap and `updatedAt` in particular is load-bearing for several existing queries (`.sort({ updatedAt: -1 })` on dashboard/progress lists).

**Soft-delete (`isDeleted: Boolean` + `deletedAt: Date`) is opt-in, not universal — apply the rule below, don't copy-paste the fields onto everything:**

> Add soft-delete when **(a)** other documents hold a live reference to this one that would dangle on a hard delete, **and** **(b)** an admin might plausibly want to restore it. Skip it otherwise.

| Has soft-delete | Doesn't |
|---|---|
| `Knowledge`, `Category`, `Company`, `Resource` — all referenced from elsewhere (`relations[]`, `Knowledge.category`, `Knowledge.companies`, `Knowledge.resources`) and admin-curated | `User` (`isActive: false` is a deactivation flag instead — a person isn't "restored," their account is reactivated); `UserProgress`, `Annotation`, `Activity`, `Attachment` — all personal or append-only data with no second document pointing at them that would dangle |

**Discriminators** for anything that's a *variant of the same entity*, never a parallel collection — this is how `Knowledge` stays one collection across `concept` / `dsa` / `interview` / `project` (`06-database-design.md` §4 owns the full field list; the *rule* here is: subtype-only fields go in the discriminator's own schema, shared fields stay on the base schema, and a discriminator schema never redeclares a base field):

```js
export const DsaKnowledge = Knowledge.discriminator(
    "dsa",
    new Schema({ pattern: String, complexity: { time: String, space: String } /* ... */ }, { _id: false })
);
```

**Embedded subdocuments are always declared `{ _id: false }`** — every subdocument schema in the codebase does this (`codeExampleSchema`, `mistakeSchema`, `relationSchema`, `providerSchema`, `revisionHistorySchema`, and every other array-of-object field). The rule behind the rule: if a piece of data needs to be addressed or mutated *independently*, by its own id, from its own route, it doesn't belong embedded at all — it belongs in its own top-level collection instead. That's precisely why `Annotation`, `Resource`, and `Attachment` are separate collections rather than arrays on `Knowledge`: each needs its own `PATCH /annotations/:id` / `DELETE /resources/:id`. If you're reaching for `{ _id: true }` on a subdocument, that's usually a sign the data wants to be its own collection instead — reconsider before overriding the default.

**Enums are imported from `constants.js`, never inlined** — in the schema *and* in its matching Zod validator, from the same constant. `RELATION_TYPES`, `KNOWLEDGE_STATUS`, `HIGHLIGHT_COLORS`, and every other enum are each defined exactly once, so a Mongoose schema enum and its Zod validator counterpart can never silently drift apart.

**Index declarations go at the bottom of the schema file**, after the schema is constructed, before `mongoose.model(...)` is called — every model file in the repo follows this layout; keep new ones consistent so indexes are always found in the same place.

## 5. Validators (Zod)

One file per writable resource in `validators/`, exporting a `create<Resource>Schema` / `update<Resource>Schema` pair (read-only resources like `search` and `dashboard` have no validator file at all — don't add one speculatively).

The update schema is derived from the create schema wherever the fields genuinely overlap, rather than redeclared from scratch:

```js
// knowledge.validator.js — `type` is immutable after create, so it's the one field
// explicitly omitted before every other field is made optional for PATCH semantics.
export const updateKnowledgeSchema = createKnowledgeSchema.omit({ type: true }).partial();
```

`validate(schema, source)` (`middlewares/validate.middleware.js`) **replaces** `req[source]` with the parsed, coerced, defaulted output of `schema.safeParse(...)` — controllers only ever see already-validated data and must never re-validate or re-trust raw `req.body` themselves.

## 6. REST Route Naming Conventions

Base paths are plural collection nouns for genuinely list-like resources (`/categories`, `/companies`, `/resources`, `/annotations`, `/activities`, `/users`) and singular mass-nouns for resources that read as "the current state of X" rather than "a list of X" (`/knowledge`, `/progress`, `/search`, `/dashboard`). The full endpoint-by-endpoint contract is `07-api-design.md`; this section is the naming *pattern*, not the list.

**Nested action sub-resources** are allowed for operations that aren't a plain CRUD verb — `POST /knowledge/:id/publish`, `GET /knowledge/:slug/related`, `POST /progress/:knowledgeId/revision`, `PATCH /progress/:knowledgeId/revision/mark`. Everything else stays noun + HTTP verb; don't add a `POST /knowledge/:id/update` when `PATCH /knowledge/:id` already means that.

**Static segments are always declared before dynamic ones in the same router** — this is a correctness rule, not a style preference (`08-backend-architecture.md` §6 has the full failure mode). Concretely:

```js
// progress.routes.js — correct order
router.get("/revision/due", getDueForRevision);
router.get("/bookmarks", getBookmarks);
router.get("/:knowledgeId", getProgress);   // must come after the routes above
```

**Slug-addressed public reads, ID-addressed admin writes** — `GET /knowledge/:slug` (bookmarkable, SEO-friendly) vs. `PATCH /knowledge/:id` (the admin UI always already has the full document, and an ID is unambiguous and immutable where a slug can change). Full rationale: `07-api-design.md` §0.

**Middleware order on a route is always** `verifyJWT` → `verifyRole(...)` → `validate(schema)` → `upload.single(...)` (only on multipart routes) → controller. Guard the whole router with `router.use(verifyJWT)` when every route needs a session (`progress`, `annotation`, `user`, `dashboard`, `activity`); guard per-route when a router mixes public reads with admin writes (`category`, `company`, `knowledge`, `resource`).

## 7. Git Commit Message Convention

Format: `type(scope): imperative-mood summary`, scope optional but preferred once a commit is scoped to one area:

```
feat(frontend): integrate shadcn/ui with base-ui and add first components
fix(auth): handle refresh-token hash mismatch as a revoked session
docs: add backend architecture and coding standards
```

- **Types**: `feat`, `fix`, `docs`, `refactor`, `chore`, `test`, `perf`, `style`, `build`, `ci` — matches Conventional Commits.
- **Imperative mood** — "Add", "Fix", "Update," not "Added" / "Fixed" / "Updates."
- **Scope** names the area touched: `frontend`, `backend`, `auth`, `knowledge`, `admin`, `search`, `docs`, etc. Omit it only for a commit that's genuinely repo-wide.
- **Body**: bullet points for anything with more than one logical change, one bullet per change — not a prose paragraph. Real example from this repo's history:
  ```
  Add backend API (auth, knowledge engine, revision, search) + core planning docs

  - Google/GitHub OAuth via Passport with JWT access/refresh cookie auth, no password login
  - Unified Knowledge Card model via Mongoose discriminators (concept/dsa/interview/project)
  - UserProgress and Annotation as separate per-user collections
  ```
- Reference a specific doc file in the body when a commit changes something that doc claims to specify (e.g. "matches docs/07-api-design.md") — makes it obvious which doc to update alongside the code, or which doc is now stale.
- `Co-Authored-By: <Agent Name> <noreply@anthropic.com>` trailer on any commit produced by an AI coding agent — already this repo's practice; keep it going.

## 8. Backend Formatting — Prettier

`backend/.prettierrc`, verbatim, is the standard — apply exactly these values, don't override per-file:

```json
{
    "singleQuote": false,
    "bracketSpacing": true,
    "tabWidth": 4,
    "semi": true,
    "trailingComma": "es5"
}
```

In practice: **double quotes**, **4-space indentation**, **semicolons required**, and trailing commas wherever ES5 already permits them — array and object literals — but **not** after the last argument in a function call or parameter list (that distinction is what separates Prettier's `"es5"` setting from its `"all"` setting; don't add a trailing comma after a final function argument even though Prettier would allow it under `"all"`). `.prettierignore` excludes `node_modules`, `.env*`, `.vscode`, `.dist`.

There is **no ESLint in `backend/`** today — no config file, no dependency. Prettier is the only enforced style tool on the backend; correctness is caught by code review and this document's conventions, not a linter. `package.json` has no `format` / `format:check` script yet either — `npx prettier --write backend/src` is the manual equivalent until one's added. Adding `"format": "prettier --write ."` and `"format:check": "prettier --check ."` to `backend/package.json` is a small, low-risk improvement worth making the next time anyone is in that file.

## 9. Frontend Formatting — ESLint + the De-Facto Prettier Profile

`frontend/eslint.config.js` is a flat config built from `@eslint/js` recommended rules, `eslint-plugin-react-hooks`'s flat recommended set (catches stale-closure / exhaustive-deps bugs — the class of bug hooks are most prone to), and `eslint-plugin-react-refresh`'s Vite preset (flags anything that would break Fast Refresh, e.g. a file mixing a component export with a non-component export). Browser globals, JSX enabled. Run via `npm run lint` (`eslint .`).

**There is no `frontend/.prettierrc` today**, and it shows: the untouched Create-Vite scaffold files (`src/main.jsx`, `vite.config.js`, `eslint.config.js`) still carry Vite's own defaults — single quotes, no semicolons, 2-space indent. Every file added since — everything under `components/`, `store/`, `lib/`, `src/pages/`, `src/routes/`, and `src/App.jsx` itself (rewritten to add routing) — has already converged on the **backend's** profile instead: double quotes, semicolons, 4-space indent. That convergence is the standard this document is codifying, not a coincidence to leave alone:

**Decision: add `frontend/.prettierrc`, identical to `backend/.prettierrc`** (§8) — one formatting profile for the whole repo, enforced by tooling instead of "match the file next to yours." Until it's added, match the newer files (`components/`, `store/`, `src/pages/`) — double quotes, semicolons, 4-space — not `main.jsx` / `vite.config.js`. Reformat the handful of scaffold holdouts the next time you're genuinely touching them for a real change; don't do a reformat-only commit just to close the gap.

## 10. PropTypes / JSDoc in a JS (Non-TS) Codebase

**No `prop-types` package** — it isn't a dependency anywhere in `frontend/package.json`, and no component in the codebase declares a `propTypes` static. This is a decision, not a gap: with RTK Query response shapes already being the single documented contract (`07-api-design.md`), a second, hand-maintained shape declaration per component would only be a second place for that contract to drift from the first. Runtime safety at the component boundary instead comes from plain destructuring with sensible fallbacks at the point of use — `items || []`, `progress?.isBookmarked`, `tags?.slice(0, 3)` — which is already the house style throughout `components/knowledge/` and `components/admin/`. Keep doing that; don't add `prop-types`.

**JSDoc is not used as formal `@param` / `@returns` blocks anywhere today** — the house style is a short `//` comment *above* a non-obvious exported function, explaining *why*, not restating the signature:

```js
// Generates a unique slug for `model` by appending -2, -3, ... on collision.
// `excludeId` lets updates keep their own slug without colliding with themselves.
export const generateUniqueSlug = async (model, title, excludeId = null) => { /* ... */ };
```

Continue that style for anything whose *behavior* isn't obvious from its name and signature. Reserve actual `/** @param */`-tagged JSDoc for an exported `utils/` / `lib/` / `services/` function whose **parameter shape itself** isn't inferable from the call site — e.g. a function that takes one options object with several optional keys:

```js
/**
 * @param {{ page?: number, limit?: number }} query - raw req.query, not yet parsed
 * @returns {{ page: number, limit: number, skip: number }}
 */
export const parsePagination = (query) => { /* ... */ };
```

Don't add JSDoc to components or to their props — a component's prop shape is documented by how it's called plus the API contract doc upstream of it, not by a third, parallel description that can silently go stale.

**`@types/react` / `@types/react-dom` are dev dependencies purely for editor IntelliSense** (JSX attribute completion, hover types on `useState`, etc., via TypeScript's ability to type-check plain `.js` / `.jsx` files, wired through `jsconfig.json`'s `@/*` path mapping) — this is not an incremental adoption of TypeScript. There's no `// @ts-check` pragma anywhere, no `.ts` / `.tsx` file in the repo, and none is planned; treat those two packages as editor tooling, not a type-safety net.

## 11. Quick Reference

- Backend file: `<resource>.<layer>.js`, camelCase resource, one resource per file per layer.
- Every controller: `asyncHandler(...)` wrapper, `throw new ApiError(...)` on failure, `res.status(x).json(new ApiResponse(...))` on success — no exceptions.
- Controller-thin by default; extract to `services/` only when reused or genuinely heavy (§3).
- Every schema: `{ timestamps: true }`; soft-delete only on referenced, admin-curated content; embedded subdocs `{ _id: false }`; enums sourced from `constants.js`.
- Static route segments before dynamic ones, in every router, always.
- Backend format: double quotes, 4-space indent, semicolons, ES5 trailing commas (`.prettierrc`) — the target for frontend too, pending `frontend/.prettierrc`.
- No `prop-types`, no formal component JSDoc — destructure-with-fallback at the call site instead.
